MobiBench: Supporting OWL2 RL n-ary rules

This web page categorizes so-called n-ary rules, which refer to a finite list of elements, and present different solutions for supporting them in a rule-based system. These solutions have been implemented in MobiBench.

Types of n-ary rules

A first subset (L1) of n-ary rules enumerate one or more restrictions, each referring to a list element (#eq-diff2, #eq-diff3, #prp-adp, #cax-adc, #cls-uni). For instance, rule #eq-diff2 flags an ontology inconsistency if two equivalent elements of an owl:AllDifferent construct are found:

$$T(?x, type, AllDifferent), T(?x, members, ?y), \\ LIST[?y, ?z_1, ..., ?z_n], T(?z_i, owl:sameAs, ?z_j) \rightarrow false$$
Rule #eq-diff2

In contrast, rules from a second subset (L2) restrain all elements in a list (#prp-spo2, #prp-key, #cls-int1). As an example, rule #cls-int1 infers that y is an instance of an intersection in case it is typed by all intersection member classes:

$$T(?c, owl:intersectionOf, ?x), LIST[?x, ?c_1, ..., ?c_n],\\ T(?y, rdf:type, ?c_1), ..., T(?y, rdf:type, ?c_n) \rightarrow T(?y, rdf:type, ?c)$$
Rule #cls-int1

A third ruleset (L3) yields inferences for all list elements (#cls-int2, #cls-oo, #scm-int, #scm-uni). As an example, rule #scm-uni infers that each member class of a union is a subclass of that union:

$$T(?c, owl:unionOf, ?x), LIST[?x, ?c_1, ..., ?c_n] \rightarrow\\ T(?c_1, rdfs:subClassOf, ?c), ..., T(?c_n, rdfs:subClassOf, ?c)$$
Rule #scm-uni

Solutions

In this section, we present solutions for the different types of n-ary rules.

To support rules from (L1) and (L3), the first solution can be applied. To support rules from (L2), one of the other three solutions is needed.

List membership functions

The following two rules infer list membership; i.e., they link each list element (indicated by rdf:first) to all preceding list cells (i.e., blank nodes making up the linked list), meaning that the first cell will be directly linked to all list elements.

$$T(?l, first, ?m) \rightarrow ?T(?l, hasMember, ?m)\\ T(?l_1, rest, ?l_2), T(?l_2, hasMember, ?m) \rightarrow T(?l_1, hasMember, ?m)$$
List membership rules

Using these two rules, rules from (L1) and (L3) can be supported. As an example, using these rules, #scm-uni (L3) may be represented as follows:

$$T(?c, unionOf, ?l), T(?l, hasMember, ?cl) \rightarrow T(?cl, subClassOf, ?c)$$
Rule #scm-uni using list membership rules

Since the list membership rules link each list element (in this case, union member class) to the first list cell $?l$, the rule will yield inferences for all member classes of the union.

Instantiate n-ary rules

Based on relevant assertions found in the ontology, this solution instantiates rules from (L2). In particular, an OWL2 RL rule is instantiated for each related assertion (e.g., propertyChainAxiom for rule #prp-spo2), leading to multiple instantiated rules per original rule. In related work, this kind of instantiation (or "rule-templates") approach is applied for any n-ary rule [1] or even for all (instance-based) OWL2 RL rules [2, 3, 4]

A clear drawback of this approach that it requires a pre-processing step for each input ontology. Also, it yields an instantiated rule for each relevant assertion, potentially inflating the ruleset. On the other hand, in case no related assertions are found, this process will not add any rules to the ruleset.

Normalize (or “binarize”) the input ontology

Normalize (or "binarize") the input ontology to only contain binary versions of n-ary assertions relevant to (L2). For instance, an n-ary intersection can be converted to a set of binary intersections as follows:

$$I = C_1 \cap C_2 \cap ... \cap C_n \equiv \\ I = C_1 \cap I_1 \wedge I_1 = C_2 \cap I_2 \wedge ... I_{(n-2)} = C_{(n-1)} \cap C_n$$
Binary version of an n-ary intersection.

The binary version of rule #cls-int1 looks as follows:

$$T(?c, intersectionOf, ?x), T(?x,hasMember,?c_1), T(?x, hasMember, ?c_2 ),\\ T(?y, type, ?c_1), T(?y, type, ?c_2) \rightarrow T(?y, type, ?c)$$
Binary version of rule #cls-int1

This rule may be considered recursive, since it both references and infers the same kind of assertion (i.e., T(?y, type, ?c)). When applying this rule on a set of binary assertions $I_1, I_2, ..., I_{(n-2)}$  , the following holds. For any resource X where $\forall i, n \leq i \leq 2: type(X) = C_i$    , the rule infers $\forall i, n-2 \leq i \leq 1: type(X)=I_i$    . In doing so, the rule travels up the chain of binary intersections until it infers $type(X) = I_1$  . Combined with $type(X) = C_1$   , the rule then infers $type(X) = I$  .

It is not hard to see how this approach only works for recursive rules, since it relies on producing partial inferences to generate the final inference. Rule #prp-key is not a recursive rule, since it infers equivalence between resources but does not refer to such relations:

$$T(?c, hasKey, ?u), LIST[?u, ?p_1, ..., ?p_n], \\ T(?x, type, ?c), T(?x, ?p_1, ?z_1), ..., T(?x, ?p_n, ?z_n), \\ T(?y, type, ?c), T(?y, ?p_1, ?z_1), ..., T(?y, ?p_n, ?z_n)\\ \rightarrow T(?x, sameAs, ?y)$$
Rule #prp-key.

Consequently, this approach only works for rules #prp-spo2 and #cls-int1 from (L2). Another drawback is that, similar to instantiating the rules, it requires pre-processing for each ontology. In particular, each relevant n-ary assertion (size n) needs to be replaced by n-2 binarized versions. Further, it generates n-2 intermediary inferences for each complete, n-ary inference. Although these are sound inferences, they could be considered to "crowd" the dataset.

Replace each n-ary rule by 3 auxiliary rules

Replace each rule from (L2) by 3 auxiliary rules. Bishop et al. [5] suggested this solution for OWLIM, based on a W3C note [6]. We show the auxiliary rules for #cls-int below:

$$T(?l, first, ?c), T(?l, rest, nil), T(?y, type, ?c)\\ \rightarrow T(?y, inIntersection, ?l)\\ \\ T(?l, first, ?c), T(?l, rest, ?l_2), T(?y, type, ?c), T(?y, inIntersection, ?l_2) \\ \rightarrow T(?y, inIntersection, ?l)\\ \\ T(?x, intersectionOf, ?l), T(?y, inIntersection, ?l) \rightarrow T(?y, type, ?x)$$
Auxiliary rules

In this solution, a first auxiliary rule starts at the end of any list ($T(?l, rest, nil)$  ), and infers an intermediary assertion for the last element (cell n). Referencing this first inference, as well as its own, subsequent inferences, a second rule travels up the list structure ($T(?l, rest, ?l_2)$   ) by inferring similar assertions for cells n-1 up until cell 1. In case the first cell is related to the relevant assertion (e.g., intersection, property chain or has-key), a third auxiliary rule generates the inference for the original, n-ary assertion.

A distinct advantage of this approach is that, in contrast to instantiating the rules and binarizing the ontology, it does not rely on pre-processing. However, each "complete" inference for an n-ary assertion (size n) requires n intermediary inferences, which do not follow from OWL2 RL semantics (instead, they ensue from our custom, auxiliary rules). As such, they can be considered to "pollute" the dataset. Bishop et al. [5] internally flag these inferences so they can be skipped during query answering. Developers may want to support something similar when adopting this solution.
